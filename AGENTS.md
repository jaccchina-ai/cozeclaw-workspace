# AGENTS.md - 多技能协作指南

> 本文档定义 proactive-agent 和 self-improving-agent 的协作机制，避免重复记录和日志膨胀

---

## 🔍 职责边界

### Proactive-Agent (短期/即时)

| 维度 | 说明 |
|------|------|
| **核心职责** | 上下文持久化，防止会话丢失关键信息 |
| **存储位置** | `SESSION-STATE.md` |
| **生命周期** | 当前会话有效，会话结束后可归档 |
| **记录粒度** | 所有关键细节（决策、偏好、纠正） |

**触发条件**:
- ✏️ 用户纠正 ("It's X, not Y")
- 📍 专有名词（人名、公司、产品）
- 🎨 偏好表达（"我喜欢..."）
- 📋 决策确认（"就用X吧"）
- 🔢 关键数值（ID、URL、日期）

---

### Self-Improvement (长期/学习)

| 维度 | 说明 |
|------|------|
| **核心职责** | 从错误中学习，持续改进行为模式 |
| **存储位置** | `.learnings/LEARNINGS.md`, `ERRORS.md` |
| **生命周期** | 长期保留，定期提升或归档 |
| **记录粒度** | 有学习价值的事件（非所有事件） |

**触发条件**:
- ❌ 操作失败（命令报错、API失败）
- ✅ 用户纠正导致认知更新（非简单偏好）
- 💡 发现更好的工作方法
- 🚫 发现知识盲区

---

## 🚫 避免重复记录的规则

### 规则1: 区分"纠正类型"

| 纠正类型 | Proactive-Agent | Self-Improvement |
|---------|-----------------|------------------|
| **简单偏好** ("用蓝色") | ✅ 记录到 SESSION-STATE | ❌ 不记录 |
| **认知纠正** ("你理解错了") | ✅ 记录到 SESSION-STATE | ✅ 记录到 LEARNINGS |
| **操作纠正** ("命令错了") | ❌ 不记录 | ✅ 记录到 ERRORS |

### 规则2: 区分"决策类型"

| 决策类型 | Proactive-Agent | Self-Improvement |
|---------|-----------------|------------------|
| **即时决策** ("用这个方案") | ✅ 记录到 SESSION-STATE | ❌ 不记录 |
| **方法论决策** ("以后都这样") | ✅ 记录到 SESSION-STATE | ✅ 记录到 LEARNINGS (best_practice) |

### 规则3: 区分"错误类型"

| 错误类型 | Proactive-Agent | Self-Improvement |
|---------|-----------------|------------------|
| **临时错误** (网络超时) | ❌ 不记录 | ❌ 不记录（除非反复出现） |
| **配置错误** | ❌ 不记录 | ✅ 记录到 ERRORS |
| **理解错误** | ✅ 记录到 SESSION-STATE | ✅ 记录到 LEARNINGS (correction) |

## 📝 记录优先级

### 高优先级（必须记录）

| 事件 | 记录位置 | 原因 |
|------|---------|------|
| 用户明确纠正理解错误 | SESSION-STATE + LEARNINGS | 影响后续行为 |
| 命令/操作失败 | ERRORS | 需要修复 |
| 关键决策确认 | SESSION-STATE | 上下文依赖 |
| 发现更好的方法 | LEARNINGS | 可复用知识 |

### 中优先级（选择性记录）

| 事件 | 记录位置 | 原因 |
|------|---------|------|
| 用户偏好表达 | SESSION-STATE | 当前会话参考 |
| 临时决策 | SESSION-STATE | 当前任务相关 |
| API临时失败 | 不记录 | 无学习价值 |

### 低优先级（不记录）

| 事件 | 记录位置 | 原因 |
|------|---------|------|
| 日常闲聊 | 不记录 | 无需持久化 |
| 明显的事实 | 不记录 | 无学习价值 |
| 用户明确说"不用记" | 不记录 | 用户指令 |

## 🔄 协作流程

### 当用户纠正时

```
1. 判断纠正类型:
   ├─ 简单偏好 → 只记录 SESSION-STATE.md
   ├─ 认知纠正 → 记录 SESSION-STATE.md + LEARNINGS.md
   └─ 操作纠正 → 只记录 ERRORS.md

2. 写入顺序:
   ├─ 先写入 SESSION-STATE.md（即时生效）
   └─ 后写入 LEARNINGS.md（异步记录）

3. 检查是否重复:
   └─ 搜索已有条目，如有则用 See Also 链接
```

### 当发生错误时

```
1. 判断错误类型:
   ├─ 临时性错误 → 不记录
   ├─ 配置/代码错误 → 记录 ERRORS.md
   └─ 反复出现 → 提升优先级

2. 记录格式:
   ├─ 简洁描述问题
   ├─ 关键错误信息
   └─ 建议修复方案

3. 不记录到 SESSION-STATE.md（错误不属于工作记忆）
```

### 当做出决策时

```
1. 判断决策类型:
   ├─ 即时决策 → 只记录 SESSION-STATE.md
   └─ 方法论决策 → 记录 SESSION-STATE.md + LEARNINGS.md

2. 记录内容:
   ├─ 决策结果
   ├─ 决策理由（简要）
   └─ 影响范围
```

## 🧹 日志维护机制

### SESSION-STATE.md 维护

| 触发条件 | 动作 |
|---------|------|
| 会话开始 | 清空过期的临时内容 |
| 任务完成 | 将重要决策归档到 MEMORY.md |
| 新任务开始 | 清空已完成的任务内容 |

### LEARNINGS.md 维护

| 触发条件 | 动作 |
|---------|------|
| 条目 > 20 条 | 审查并提升/归档 |
| 状态 = resolved | 移动到归档区 |
| 状态 = promoted | 合并到 SOUL.md/AGENTS.md/TOOLS.md |

### ERRORS.md 维护

| 触发条件 | 动作 |
|---------|------|
| 错误已修复 | 标记为 resolved |
| 条目 > 30 条 | 归档旧条目 |
| 重复错误 | 合并为单条，增加计数 |

```
事件发生
    │
    ├─ 是错误吗？
    │   ├─ 是 → 临时性？
    │   │       ├─ 是 → 不记录
    │   │       └─ 否 → 记录 ERRORS.md
    │   │
    │   └─ 否 ↓
    │
    ├─ 是用户纠正吗？
    │   ├─ 是 → 简单偏好？
    │   │       ├─ 是 → 只记录 SESSION-STATE.md
    │   │       └─ 否 → 记录 SESSION-STATE.md + LEARNINGS.md
    │   │
    │   └─ 否 ↓
    │
    ├─ 是决策吗？
    │   ├─ 是 → 方法论级别？
    │   │       ├─ 是 → 记录 SESSION-STATE.md + LEARNINGS.md
    │   │       └─ 否 → 只记录 SESSION-STATE.md
    │   │
    │   └─ 否 ↓
    │
    └─ 其他 → 不记录
```

## ✅ 最佳实践

### DO ✅

1. **记录前先搜索** - 避免重复记录
2. **简洁记录** - 一句话说明问题
3. **及时标记状态** - resolved/wont_fix
4. **定期提升** - 将成熟模式写入 SOUL.md

### DON'T ❌

1. **不要记录每个细节** - 只记录有价值的
2. **不要重复记录** - 一个事件只记录一次
3. **不要无限保留** - 定期清理归档
4. **不要混合层级** - 临时 vs 永久要分开

## 🧠 MoA 增强版分析调用规则

### 核心原则

**当遇到复杂问题时，调用 MoA (Mixture of Agents) 进行多模型协作分析，获取更全面、更可靠的解决方案。**

### 触发场景

| 场景类型 | 具体情况 | 优先级 |
|---------|---------|--------|
| **受阻场景** | Task 功能实现受阻 | 🔴 高 |
| | API 限制/配额问题 | 🔴 高 |
| | 逻辑难题/架构决策 | 🔴 高 |
| | 第三方集成障碍 | 🟡 中 |
| **复盘场景** | Task 定期复盘 | 🟡 中 |
| | 寻求功能增强方向 | 🟡 中 |
| | 性能优化分析 | 🟡 中 |

### MoA 输出格式要求

调用 MoA 时，要求输出包含以下结构：

```markdown
## 问题根因分析
[深入分析问题的根本原因，而非表面现象]

## 可行方案 (3~5个)

### 方案 1: [方案名称]
- **优点**: ...
- **缺点**: ...
- **适用场景**: ...

### 方案 2: [方案名称]
- **优点**: ...
- **缺点**: ...
- **适用场景**: ...

...

## 推荐实施步骤
1. [第一步]
2. [第二步]
3. [第三步]
...
```

### 调用流程

```
问题识别
    │
    ├─ 判断是否属于触发场景
    │   ├─ 是 → 调用 MoA
    │   └─ 否 → 常规处理
    │
    ├─ MoA 分析 (多模型协作)
    │   ├─ Layer 1: Proposers (3-4 模型生成方案)
    │   ├─ Layer 2: Reviewers (2-3 模型评审)
    │   └─ Layer 3: Synthesizer (综合输出)
    │
    ├─ 输出结构化报告
    │
    └─ 人类决策确认
        ├─ 批准 → 执行
        └─ 调整 → 重新分析
```

### 与其他 Skills 的协作

| Skill | 协作方式 |
|-------|---------|
| **proactive-agent** | 记录 MoA 分析结果到 SESSION-STATE.md |
| **self-improvement** | 从 MoA 分析中提取学习点到 LEARNINGS.md |
| **unifuncs** | 深度研究任务可调用 MoA 增强分析 |
